# 適用範囲

Delphi Method は RAND Corporation が 1950 年代に技術予測のために開発した手法。匿名性により追従バイアス・権威バイアス・声の大きさバイアスを構造的に排除します。AI agent への適用としては、各 agent が「誰の発言か知らないまま意見集約だけを参照する」設計にすることで、agent 間の引きずられ（特にエコーチェンバー）を防ぎます。

向いている問題:
- 未来予測（数値・シナリオ・確率）
- 専門家の知識集約が必要な評価
- 複数の独立判断が必要な優先順位付け
- 「誰が言ったか」が判断を歪めやすい議題

向いていない問題:
- 正解が一意に決まる事実問題（multi-agent-debate が良い）
- 役割分担で深掘りすべき提案（hegelian / red-blue が良い）

# チーム構成

teammate を N 個並列 spawn する（既定: N=4、複雑な質問なら N=6〜8）:

- **delphi-expert**（subagent: `delphi-expert`）: 各ラウンドで独立に意見を出す。他 expert の発言は誰のものか分からない匿名集約のみ参照

`tools` は subagent definition で `Read, Grep, Glob` のみに制限済み。

# 議論プロトコル（Delphi Method）

## Round 0: 質問の確定（lead 単独）

質問を読み、以下を確定:

1. **質問の明確化**: 何を予測・評価・優先順位付けするか具体化。曖昧さは lead が補完
2. **回答形式の指定**: 数値（信頼区間込み）/ 質的シナリオ / 順位リスト / 確率 など、回答形式を明示
3. **N の決定**: 既定 N=4。専門性の幅が必要なら N=6〜8。多すぎると集約が散漫になる

## Round 1: 独立意見（delphi-expert を N 個並列 spawn）

N 個の delphi-expert を **並列 spawn**:

- 各 spawn prompt は **完全に同一**（同じ質問、同じ回答形式）
- 各 expert は他 expert が誰かを知らない
- 互いの意見は **絶対に見せない**

各 expert の Round 1 出力（意見 + 信頼区間 + 根拠 + 確信度 + 留保）を集める。

## Round 2: 匿名集約とフィードバック（lead 単独 → expert 並列 spawn）

**lead が集約を作成**（匿名性厳守）:

1. **発言者を完全に秘匿** する。「expert-1 が...」「Aさんが...」のような attribution 禁止
2. **意見の分布**: 中央値・幅・四分位範囲・外れ値（数値の場合） / 主要な立場の分類（質的の場合）
3. **主要な対立軸**: 何が意見を分けているか（前提の違い / 根拠データの違い / 価値観の違い）
4. **論拠の集約**: 各立場の主要な論拠を匿名で列挙（「ある expert は X と論じている」のように発言主を伏せる）
5. **欠けている観点**: 全 expert がカバーしていない側面があれば lead が指摘

集約結果を全 delphi-expert に **並列再 spawn** で渡し、各自に意見を再考させる:

- 各 expert は集約を読み、意見を維持/修正/撤回する
- 自分自身の Round 1 発言も匿名集約の一部として混ざっているため、誰が誰の意見か分からない

## Round 3 以降: 収束まで繰り返し（最大 4 ラウンド）

Round 2 の結果を再集約し、再フィードバックする:

**収束判定**:
- **数値収束**: 中央値 ± 四分位範囲が前ラウンドから安定（例: 四分位範囲が 20% 以上縮小しない）
- **質的収束**: 主要な立場の数が安定し、各 expert が立場を変えない
- **健全な対立で安定**: 意見が割れたまま安定し、各 expert が「これ以上変えない」と表明
- **未収束**: 各ラウンドで分布が動き続けている → さらにもう 1 ラウンド（最大 4 ラウンドまで）

## Final: 統合（lead 単独）

各 expert の最終意見を統合して結論を出す。

1. **収束分布の提示**: 数値なら中央値・信頼区間、質的なら主要な立場の分布
2. **lead としての判定**:
   - **強い収束**: 意見が一致したので採用
   - **収束だが幅あり**: 中央傾向を採用するが幅を明示
   - **健全な対立**: 単一答えに収束しない理由を明示し、立場ごとの妥当性を併記
   - **未収束**: 4 ラウンドで収束しなかった旨と、追加で必要な情報を明示
3. **少数派・外れ値の扱い**: 少数派の論拠が論理的に強かった場合は明示し、結論の留保とする
4. **匿名性の維持**: 最終出力でも「expert-1 が」などの attribution を一切しない（個別の論拠は「ある立場の論拠」として匿名提示）
5. **追従バイアスの自己検証**: ラウンドを通じて意見が動いた方向が、本当に論理に基づくものか、エコーチェンバーではないかを lead が点検

## lead の進行原則

- **匿名性は神聖**: 一度でも attribution を漏らすと Delphi の効果が消失する。集約・最終統合の全段階で氏名・番号を使わない
- **意見を動かそうとしない**: lead は集約の中立提示に徹する。誘導禁止
- **少数派意見を捨てない**: 多数決ではない。少数派が論理的に強ければ結論に反映する
- **収束を急がない**: 早期収束は追従バイアスの可能性。各ラウンドで本当に論理が動いたか点検
- **集約の質が結論の質**: 数値分布だけでなく論拠の質も集約に含める
- 議論作業は読み取り中心なのでファイル衝突は通常起きないが、書き込み作業を割り振る場合は同一ファイル編集を避けるよう task を切る

# 他形式との位置づけ

- **multi-agent-debate**: 対称な solver が **直接議論**。誰の発言か可視。追従バイアスのリスクあり
- **delphi**: 完全匿名集約。**追従バイアス排除を最優先**。直接議論なし、ラウンド制でゆっくり収束
- 同じ問題を multi-agent-debate と delphi の両方で回すと結果が変わるはずで、その差が追従バイアスの規模を示唆する（実験的に興味深い）
