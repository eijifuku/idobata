$ARGUMENTS

# 重要な前提

これは **発散プロトコル** であり、収束プロトコル（hegelian / red-blue / pugh-matrix 等）ではない。

- **判定・採否を出さない**（出力は「アイデアの分布」であり「結論」ではない）
- **critic 役を置かない**（早期淘汰がアイデアを殺す）
- 出力は後段の収束プロトコル（pugh-matrix / rational-choice / red-blue）に渡す前提

# チーム構成

teammate を **N = 5〜8 名** spawn する。全員 subagent: `ideator`。

各 ideator には **互いに直交する lens** を 1 つ（または複合）割り当てる。lens 設計が本プロトコルの肝。

# プロトコル

## Step 0: 議題のフレーミング（lead 単独）

議題を読み、以下を明確化:

- **発散の対象**: 何のアイデアを欲しているか（例: 機能案 / マーケティング施策 / 命名 / 解決策 / イベント企画）
- **質問形に変換**: "How might we ___?" 形式に書き直す（Stanford d.school 流）
- **既知の凡庸解**: 既に思いついている当たり前の答えを 3〜5 個列挙し、ideator にはこれを避けるよう指示

## Step 1: lens 設計（lead 単独・最重要）

N 個の ideator に割り当てる lens を **意図的に異質に** 設計する。良い lens セットの条件:

- **直交性**: 同種の lens を複数置かない（ペルソナ 5 個より、ペルソナ 2 + 制約 1 + 類推 1 + 反転 1）
- **強さ**: 弱い lens（「クリエイティブに考えて」）は禁止。具体物を指定する
- **議題からの距離**: 半分は議題ドメイン内、半分は完全に外（生物学・料理・物流など）
- **少なくとも 1 つは反転 lens**（worst possible idea）を含める

### 議題タイプを問わず使える lens 構成テンプレート

汎用的な構成（5〜6 lens 推奨）:
- **ペルソナ lens** ×2: 議題の想定外当事者を 2 種（年齢・職業・文化・専門性が遠い人）
- **制約 lens** ×1: 強い制約を 1 つ（リソース極小 / 時間極短 / 1 要素のみ使用 等）
- **類推 lens** ×1: 全く別ドメインから手法・構造・比喩を借りる
- **反転 lens** ×1: 「最悪のアイデアは?」「目的を真逆に取ったら?」
- **ランダム刺激 lens** ×1（任意）: 議題と無関係な単語・概念を強制的に絡める

### 議題タイプ別の lens 例（あくまで例、絶対ではない）

**プロダクト・機能案**: 想定外ユーザ 2 / リソース極小制約 / 別業界の同種問題類推 / 「ユーザに最悪な体験は?」反転

**問題再定義・リフレーミング**: 異なる利害関係者ペルソナ / 時代 lens（100 年前 / 100 年後）/ 「この問題が解決されない方が良い理由は?」反転

**自然・科学的想像（生物・宇宙・現象の探索）**: 物理制約 lens / 異化学・異物理 lens / 既知系からの類推 / 進化時間スケール lens / ランダム刺激

**イベント・場づくり**: 想定外参加者ペルソナ / 「会議室なし」「お金なし」制約 / 別ジャンル（ライブ・ゲーム・宗教儀式）からの類推 / 「最悪の会にする方法は?」反転

**命名・コピー**: 異言語・異文化ペルソナ / 1 音節 or 長文の制約 / 「最悪の名前は?」反転 / 別商品分野からの類推

**戦略・施策**: 異業界 CEO ペルソナ / 予算ゼロ制約 / 「失敗するには?」反転 / 別時代の戦略類推

## Step 2: 並列 spawn（Round 1）

N 個の ideator を **同時並列** で spawn。各 spawn prompt に含めるべきもの:

- 議題本文（"How might we" 形式）
- 既知の凡庸解（避けるべき例として）
- **その ideator 専用の lens**（他 ideator の lens は教えない）
- 出力フォーマットの指定（agent 定義通り）

## Step 3: アイデア集約とクラスタリング（lead 単独）

全 ideator の出力を集めたら lead が:

1. 全アイデアを **1 つのフラットリスト** にまとめる（出典 ideator を併記）
2. 重複・近似案を **マージ**（ただし lens 由来の微妙なニュアンス差は残す）
3. **クラスタリング**: 5〜10 のクラスタに分類。各クラスタに名前を付ける
4. 各クラスタから **wildcard** を 1 個選ぶ（最も突飛 / 想定外だったもの）

## Step 4: "Yes-and" 拡張（Round 2、任意）

時間と価値があれば実施。各 ideator を **同じ lens のまま** 再 spawn し、以下を渡す:

- Round 1 の全アイデアリスト（クラスタ済み）
- 「他人のアイデアから 1 つ選び、自分の lens で **発展（yes-and）** させた案を 5 個出せ」
- 否定的批評は引き続き禁止

## Step 5: SCAMPER 変形（Round 3、任意）

各 wildcard に対し **lead 自身が SCAMPER 7 操作** を順に当てて派生案を出す。
（独立 skill `idobata:scamper` を呼ぶ選択肢もある）

## Step 6: 最終出力（lead）

```
# 発散結果: <議題の "How might we" 形式>

## 概要
- 総アイデア数: N
- クラスタ数: M
- lens 構成: <使った lens を列挙>

## クラスタ別ハイライト
### クラスタ 1: <名前>
- 代表案 3〜5 個（番号 + 1 文）
- このクラスタ内 wildcard: <突飛な案>

### クラスタ 2: ...

## 全アイデア一覧（番号 + 案 + 出典 lens）
1. ... [lens=ペルソナ:漁師]
2. ...
（全 N 案）

## 次の収束ステップ提案
- 例: 「上位クラスタを候補に /idobata:pugh-matrix で datum=現状 として比較せよ」
- 例: 「特定クラスタを /idobata:rational-choice で重み付き評価せよ」
- 例: 「特定の 1 案を /idobata:red-blue で攻撃検証せよ」
```

# 重要な制約

- lead は **アイデアの良し悪しを判定しない**。クラスタリング・整理のみ。
- 「実用的な」順に並び替えない。lens の多様性を保つ。
- ideator が出力フォーマットを守らなかった場合のみ再 spawn する（内容での再 spawn は禁止）。
- 議題が「正解のある問題」（数学・コード正誤など）の場合は本プロトコルを使わず `/idobata:multi-agent-debate` を勧める。
- 議題が「複数候補からの選定」の場合は `/idobata:pugh-matrix` か `/idobata:rational-choice` を勧める。
