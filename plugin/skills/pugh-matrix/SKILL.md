# 適用範囲

Pugh Matrix は Stuart Pugh (1981) がエンジニアリング設計選定のために提案した手法。rational-choice の親戚ですが性格が異なります:

- **datum（基準候補）との相対比較**: 絶対スコアではなく +/-/S の 3 値
- **採点バイアス排除**: 中央値偏重・スケール解釈差・ハロー効果が起きにくい
- **反復改良が本質**: 弱点候補を他候補の長所で改良した A' を生成し、次世代として再評価
- **軽量**: rational-choice より少ない情報で回せ、繰り返し使える

向いている問題:
- 技術選定（DB / フレームワーク / アーキテクチャパターンの比較）
- 設計案 A/B/C の初期スクリーニング
- 繰り返し改良したい比較（候補が固定でなく成長する）
- 「現状維持」を datum にして変更の価値を測る場面

向いていない問題:
- 重み付け・絶対スコアが本質的に重要な意思決定（rational-choice）
- 候補が 1 つしかない採否判定（red-blue）
- 価値観の対立を統合したい（hegelian）

# チーム構成

teammate を必要に応じて複数回 spawn する:

- **pugh-comparator**（subagent: `pugh-comparator`）: datum vs 候補 X を軸ごとに +/-/S 判定。**候補ごとに並列 spawn**
- **pugh-improver**（subagent: `pugh-improver`）: 弱点候補を他候補の長所で改良し A' を生成

`tools` は subagent definition で `Read, Grep, Glob` のみに制限済み。

# プロトコル（Pugh Matrix）

## Round 0: 候補・datum・軸の確定（lead 単独）

選択肢を読み、以下を確定:

1. **候補リスト**: 比較する選択肢を 3〜6 個に確定（datum 含む）
2. **datum の選定**: 基準候補を 1 つ選ぶ。原則は以下のいずれか:
   - **現状維持**: 「今使っているもの」「既存実装」を datum にすると、変更の価値が測れる
   - **業界標準**: 最も馴染みのある選択肢
   - **最も well-known な候補**: 全員が知っていて評価しやすいもの
3. **評価軸**: lead が直接 5〜10 個の軸を提案する（lens 概念は Pugh にはない、フラットな技術視点で OK）。例:
   - 技術選定: 学習コスト / 性能 / 保守性 / コミュニティの規模 / セキュリティ / コスト / 既存技術との親和性
4. **候補・datum・軸をユーザーに提示**して進めるか確認する

軸提案で迷う場合は、過去の rational-choice 形式から criteria-proposer を借りる手もあるが、Pugh は軽量さが売りなので lead 直接が原則。

## Round 1: 全候補を並列比較（pugh-comparator 並列 spawn）

datum 以外の各候補について pugh-comparator を **並列 spawn**:

- 各 spawn prompt に: datum 名と内容、対象候補名と内容、評価軸（定義込み）
- 各 comparator は独立に +/-/S 判定
- datum 自身は全軸 S（自己比較）として扱う

## Round 2: 集計と弱点特定（lead 単独）

全 comparator の結果を集約:

1. **マトリクス表示**:
   ```
   | 軸 \ 候補 | datum (基準) | A | B | C |
   | --- | --- | --- | --- | --- |
   | 軸1 | S | + | - | S |
   | 軸2 | S | - | + | + |
   | ... | ... | ... | ... | ... |
   | + 合計 | 0 | n_A | n_B | n_C |
   | - 合計 | 0 | m_A | m_B | m_C |
   | 単純差 | 0 | n_A-m_A | n_B-m_B | n_C-m_C |
   ```

2. **弱点候補の特定**: 単純差が低い、または特定軸で - が多い候補を flag
3. **強い候補の特定**: 単純差が大きく + に振れた候補
4. **改良の余地評価**: 弱点候補の中で、改良で datum を超えうるものを 1〜2 個選ぶ

## Round 3: 改良候補の生成（pugh-improver spawn、最大 2 反復）

Round 2 で選んだ弱点候補について pugh-improver を spawn:

- 各 spawn prompt に: 改良対象候補、Round 2 の完成マトリクス全文、他候補（+ を取っている軸の参考用）
- improver は A → A' のような改良候補を生成

## Round 4: 改良候補を含めて再比較（pugh-comparator 並列 spawn）

改良候補（A' など）を新しい比較対象として、Round 1 と同じく pugh-comparator を並列 spawn し、datum との比較を行う。

**反復継続判定**:
- **改良候補が datum を超えた**: 単純差が大きくプラスになった → datum 交代を検討（次反復で A' を新 datum に）
- **改良が部分成功**: 一部軸が - → S/+ になったが datum を超えない → さらに改良（最大 2 反復）
- **改良不能**: improver が「これ以上改良できない」と判断 → 反復終了
- **反復は最大 2 回まで**。それ以上は収益逓減

## Final: 推奨候補と決定（lead 単独）

最終マトリクスから推奨を出す:

1. **最終マトリクスの完全表示**: 全候補（改良候補含む）× 全軸の +/-/S 表
2. **推奨候補**: 単純差が最大、かつ critical な - を持たない候補を選ぶ
3. **datum 維持 vs 交代**:
   - 推奨候補が datum を明確に超える（複数軸で +、致命的 - なし）→ **新 datum として推奨**
   - datum を超える候補がない → **datum 維持を推奨**（変更の価値なし）
4. **採用しなかった候補の評価**: なぜ採用しなかったかを軸ベースで説明
5. **改良の履歴**: 反復で何を学んだか（A → A' で何が改善され何が悪化したか）
6. **撤退条件**: この決定を将来見直すべき状況（軸の重要度が変わった、新しい候補が出てきた、評価軸自体が陳腐化した、など）

## lead の進行原則

- **datum 選択を恣意的にしない**: 弱い候補を datum にすると他候補が容易に + を取り、結論が誇張される。「現状維持」または「業界標準」を datum にするのが王道
- **+/-/S の単純合計を絶対視しない**: 1 つの critical な -（致命的弱点）がある候補は、+ を多く取っても採用すべきではない。non-compensatory 軸の概念は Pugh にも適用する
- **反復改良の価値を意識する**: Pugh Matrix の真価は採点ではなく **A → A' → A'' の世代更新**。1 回の比較で結論を急がない
- **datum 交代を恐れない**: 改良候補が datum を超えたら datum を交代させて続行。これが「世代を重ねる候補進化」になる
- **軽量さを保つ**: rational-choice より軽く回せるのが Pugh の利点。重み付けや感度分析を持ち込みすぎない（必要なら rational-choice に切り替える）

# 他形式との位置づけ

- **rational-choice**: 重み付き絶対スコアの精緻な比較。最終決定段階向き
- **pugh-matrix**: 軽量な相対比較 + 反復改良。**初期スクリーニング・繰り返し検討向き**
- **red-blue**: 1 候補の採否判定
- **hegelian**: 意見対立の統合

技術選定の典型ワークフロー: **pugh-matrix で候補を絞り込み・改良 → 最終 2〜3 候補に rational-choice で精密判定**
