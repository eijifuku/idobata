以下のお題（既存物）を SCAMPER で変形し、改良案・派生案を大量生成してください。あなた（このメインセッション）が **lead = ファシリテーター（お題分解 + 変形案集約役）** です。

# お題（変形元の既存物）

$ARGUMENTS

# 重要な前提

これは **既存物の変形に特化した発散プロトコル**。ゼロから発想する場合は `/idobata:brainstorm` を使う。

- **判定・採否を出さない**（後段の収束プロトコルに渡す前提）
- **critic 役を置かない**
- **lead はお題の本質分解とクラスタリングに集中**

# チーム構成

teammate を **7 名** spawn する。全員 subagent: `transformer`。

各 transformer に SCAMPER 7 操作のうち 1 つを 1:1 で割り当てる:

- `transformer-S`: Substitute
- `transformer-C`: Combine
- `transformer-A`: Adapt
- `transformer-M`: Modify / Magnify / Minify
- `transformer-P`: Put to other use
- `transformer-E`: Eliminate
- `transformer-R`: Reverse / Rearrange

（subagent type は同じ `transformer` で、prompt で操作を指定する）

# プロトコル

## Step 0: お題の本質分解（lead 単独）

お題を以下の観点で分解し、全 transformer に共有する **共通参照** を作る:

- **構成要素**: お題が何で出来ているか（モノ / 機能 / 手順 / 関係 / 前提条件）
- **目的・効用**: お題が達成しようとしていること
- **暗黙の前提**: 「当然こうあるべき」とされているが疑える前提

本質分解の例（任意ドメイン）:
- ソフトウェア機能: 機能 = {ユーザ操作 + 入力 + 内部処理 + 出力 + 副作用 + 失敗時挙動}
- 物理プロダクト: 製品 = {素材 + 形状 + 機構 + 接合 + 仕上 + 包装 + 想定使用シーン}
- 業務プロセス: プロセス = {トリガ + 入力 + 担当 + 工程順序 + 引き継ぎ + 完了条件}
- イベント: イベント = {目的 + 参加者 + 場所 + 時間構成 + 進行役 + 成果物}

## Step 1: 7 並列 spawn

7 個の transformer を **同時並列** spawn。各 spawn prompt に含めるもの:

- お題本文
- Step 0 の本質分解結果
- **その transformer の担当操作**
- 出力フォーマット指定（agent 定義通り）

## Step 2: 集約とクラスタリング（lead 単独）

7 transformer の出力（合計 56〜84 案）を集めたら:

1. 操作別のセクションをまず保持して全案リスト化（出典操作併記）
2. **クロス操作で似た発想に至った案** をマージ（操作の相補性が見える）
3. **クラスタリング**: 7〜10 のクラスタに再編成（操作軸ではなく「変形の方向性」軸で）
4. 各クラスタの **wildcard**（最も突飛 / 想定外）を 1 個ずつ抽出

## Step 3: 二段 SCAMPER（任意）

特に有望な変形案 1〜2 個に対し、**さらに 7 操作を当てる** 二段階適用が可能。
1 段目で出た案を新たな「お題」として `/idobata:scamper` を再帰呼び出しするか、lead 単独で簡易実施する。

## Step 4: 最終出力

```
# SCAMPER 変形結果

## お題
<元お題 1〜2 文>

## 本質分解
- 構成要素: ...
- 目的: ...
- 暗黙の前提: ...

## 操作別ハイライト
### S — Substitute
- 代表案 2〜3 個
- 効いた軸: ...

### C — Combine
...

（7 操作すべて）

## クラスタ別ビュー
### クラスタ 1: <変形の方向性名>
- 案 3〜5 個（操作横断）
- wildcard: ...

## 全案一覧
1. [操作=S] <案>
2. [操作=C] <案>
... (合計 N 案)

## 次の収束ステップ提案
- 上位案を /idobata:pugh-matrix で datum=元お題 として比較
- 特定 1 案を /idobata:red-blue で攻撃検証
```

# 使用例

本プロトコルは「既存物」がある任意のお題に適用できる。以下は適用例の一部:

## 例 1: ソフトウェア機能の派生案
お題に既存機能を 1 つ渡す（例: 「ユーザがフォームから問い合わせを送る機能」）。
- S: 入力をフォーム → 音声 / チャット / 画像 OCR に置換
- C: 問い合わせ + FAQ 自動検索 / + 翻訳 を結合
- A: チケット管理ツールの workflow を adapt
- M: 入力項目数を最小化 / 最大化 / 段階的開示
- P: 同じ機能を社内ヘルプデスクに転用
- E: フォーム自体を排除し、メンション検知に置換
- R: 受信側がプッシュで質問を聞きに行く方向に逆転

## 例 2: 物理プロダクトの改良
お題に既存製品を 1 つ渡す（例: 「ステンレス製水筒」）。
- S: 素材を竹 / 紙 / 生分解性樹脂に置換
- C: 水筒 + 浄水機能 / + 温度表示
- A: 魔法瓶構造を断熱コンテナから adapt
- M: 容量を 50ml / 5L に / 厚みを極薄化
- P: 同じ筐体を花瓶 / ペン立てに転用
- E: 蓋を排除して別構造で密閉
- R: 内外を反転（外が冷える保冷スリーブ）

## 例 3: イベントの改良
お題に既存イベントを 1 つ渡す（例: 「月 1 回・対面開催の社内勉強会」）。
- S: 場所を対面 → オンライン / 屋外 / カフェに置換 / 進行役を持ち回りに
- C: 勉強会 + 食事会 / + ハッカソン を結合
- A: TED 形式（短時間 LT 連発）を adapt
- M: 60 分 → 15 分に短縮 / 1 日合宿に拡大 / 頻度を週 1 / 半年 1 に
- P: 同じ枠を社内採用説明会 / 顧客向けセミナーに転用
- E: 発表者を排除し参加者全員の雑談だけにする / 司会を排除
- R: 参加者が事前に資料を読み、当日は質問だけする反転授業形式

## 例 4: 制度・ルールの派生
お題に既存ルールを 1 つ渡す（例: 「会議は事前にアジェンダを共有する」）。
- S: 共有手段をドキュメント → 動画 / 音声に置換
- C: アジェンダ + 結論ドラフト を事前共有
- A: 学術カンファレンスの abstract 制度を adapt
- M: アジェンダを 1 行に / 詳細化
- P: 同じルールを 1on1 にも転用
- E: アジェンダ無しで開始、最初 5 分で全員が議題を出す
- R: 会議後にアジェンダを書く（決定の事後文書化）

# 重要な制約

- 7 操作のうち 1 つを skip しない（直交性が崩れる）
- transformer に複数操作を渡さない（1 spawn = 1 操作）
- lead は変形案を判定しない。整理のみ。
