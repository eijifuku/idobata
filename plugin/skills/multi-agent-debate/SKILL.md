# 適用範囲

Multi-Agent Debate は Du, Li, Torralba et al. (2023) "Improving Factuality and Reasoning in Language Models through Multiagent Debate" の手法に基づき、**正解が存在する問題** や **検証可能な推論** で精度向上が報告されています:

- 数学・論理パズル
- ファクトチェック
- コード推論・バグ検出
- 多段推論を要する常識問題

主観的価値判断・嗜好の問題には別形式（hegelian, red-blue, six-hats）が向きます。

# チーム構成

teammate を N 個並列 spawn する（既定: N=3、複雑な問題なら N=5）:

- **solver**（subagent: `solver`）: Round 1 で独立に解答、Round 2 以降は他 solver の解答全文を見て再解答

`tools` は subagent definition で `Read, Grep, Glob` のみに制限済み。

# 議論プロトコル（Multi-Agent Debate）

## Round 0: 問題の確定（lead 単独）

問題を読み、以下を確定:

1. **問題の明確化**: 曖昧な部分は lead が補完。例「最大値を求めよ」なら定義域・整数 vs 実数を明示
2. **N の決定**: 既定 N=3。問題の難易度・必要な多様性から 3〜5 個に調整
3. **採点可能な答えの形式**: 数値 / 一文の結論 / 推論手順 など、最終答えの形を明示

## Round 1: 独立解答（solver を N 個並列 spawn）

N 個の solver を **並列 spawn**:

- 各 spawn prompt は **完全に同一**（同じ問題、同じ形式指示）
- 各 solver は他 solver の存在を意識するが、誰がいるかは知らない
- 互いの解答は **絶対に見せない**

各 solver の Round 1 出力（推論プロセス + 最終答え + 確信度 + 不明点）を集める。

## Round 2: 議論修正（solver を N 個並列 spawn、最大 3 周）

全 solver の Round 1 出力を **要約せず全文** で各 solver に渡し、並列再 spawn:

- 各 solver は他 N-1 人の解答を読み、自分の解答を維持/修正/撤回する
- 解答は同時に更新される（並列）

**収束判定**:
- **完全収束**: 全 solver の最終答えが一致 → Final へ
- **多数決優位**: N の過半数が一致、少数派の根拠が薄い → Final へ
- **健全な対立**: 意見が割れたまま、双方根拠が強い → さらに Round 2 を 1 周（最大計 3 周まで）
- **発散**: ラウンドを重ねるごとに意見が分散していく → Round 2 を打ち切り Final で対立を残す形で統合

## Final: 最終答え（lead 単独）

solver たちの議論を統合して最終答えを出す。

1. **収束した答え**: 全 solver または多数派が一致した最終答えを採用
2. **少数派意見の処理**:
   - 少数派の論理が **正当**: 「多数派の答えは X だが、少数派が Y を指摘した。Y を踏まえると正解は X′」のように修正する勇気を持つ
   - 少数派の論理が **不当**: なぜ却下するか明示
3. **論拠の統合**: 採用した最終答えに至る最も強い推論プロセスを書き直す（複数 solver の良い部分を組み合わせてもよい）
4. **確信度評価**:
   - 全 solver の確信度（高/中/低）の分布
   - lead としての最終確信度
5. **未解決事項**: 議論で解けなかった部分・追加情報があれば確信度が上がる部分を明示

## lead の進行原則

- **Round 1 では絶対に他者解答を見せない**: 独立性が手法の核心。これを崩すと精度向上効果が失われる
- **追従バイアス監視**: Round 2 以降で「全員が初期解答を捨てて 1 人の解答に流れる」場合は要注意。多数派の論理が本当に正当か lead が点検
- **自分は答えを先に決めない**: lead は solver たちの議論を観察する役。自分の答えを前提に議論を誘導しない
- **少数派に発見の価値を認める**: 多数派が間違っているとき、しばしば少数派が正解。多数決に頼り切らない
- 議論作業は読み取り中心なのでファイル衝突は通常起きないが、書き込み作業を割り振る場合は同一ファイル編集を避けるよう task を切る

# 他形式との位置づけ

- **hegelian / red-blue**: 役割分担型（賛成 vs 反対 / 攻撃 vs 防衛）。役割が結論を歪める可能性
- **delphi**: 匿名で意見集約。追従バイアスを完全排除する代わりに直接議論なし
- **multi-agent-debate**: **対称な solver が直接議論**。役割バイアスなし、独立性は Round 1 のみ確保。事実問題向け
